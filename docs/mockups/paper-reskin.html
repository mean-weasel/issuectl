New issue creation
Tap the FAB (mobile) or the "+ new issue" button (desktop). Title and body feel like writing in a notebook — big serif, no borders, just placeholders that fade out as you type. Repo assignment and labels are secondary fields below the divider. Save as a local draft OR assign immediately.

How it feels

  • Title is the hero. 26px serif, no border, no label. Placeholder is italic, value is upright. Tap and you're writing.
  • Body is a longer version. Same serif, 15px, multi-line. No toolbar, no markdown preview. Raw markdown until it's pushed to GitHub.
  • Repo selector is optional. Tapping it opens the same assign sheet from the main list. You can close the form entirely unassigned.
  • Labels wait. Until a repo is chosen, the labels field reads "assign a repo first" — we can't show real labels without a target.
  • Two save paths: save draft (quiet, stays local) and assign & save (opens the repo sheet, then syncs to GitHub on pick).

Desktop version

  • Same form in a 640px modal, centered over a scrim. Esc closes. Cmd+Enter saves.
  • Layout is identical — this form doesn't need to grow sideways. Writing prefers narrow columns.

Voice guidelines

  • Short, lowercase, italic for secondary copy — "a quiet place," "breathe, or draft the next thing"
  • No emoji, no exclamation marks. The paper aesthetic carries the mood.
  • Italic fragments punctuate; they don't narrate.
  • Brand signature: the word issuectl in italic serif, followed by a small green dot like a period.
  • Status chips use the accent color with soft surface for a "keeping watch" feel.
  • Error states use brick red sparingly — only when something actually broke. Expected empty states stay calm.

Settings decisions

  • One page, four sections — repos, defaults, claude config, worktree cleanup. A small "about" footer shows auth + version + data location + bound address.

Progress behavior

  • Stepped checklist instead of a generic spinner — tells you what's happening at each stage.
  • Active step pulses the inner dot in accent green. Completed steps fill and show a check. Pending steps stay faint.
  • Step details show concrete values (worktree path, branch name, file count) so you can sanity-check.
  • Same view on mobile and desktop — the form-to-progress transition happens inside the same modal/sheet, just with larger type and padding on desktop.
  • Bailout path: if any step fails, the active step turns brick red, shows the error, and offers a retry or "open logs" action. (Not mocked here — same visual vocabulary.)
flow 10
Auth error — gh not ready
Shown on app load if the gh CLI isn't installed or isn't authenticated. Blocks the app until fixed. Stays in the Paper aesthetic — brick red for the single error signal, three clear remediation steps, a retry button.
mobile · auth error
!

GitHub CLI not ready

issuectl reads your GitHub credentials from the gh CLI. It isn't installed or isn't authenticated.

1. install it: brew install gh
2. authenticate: gh auth login
3. hit retry — we'll pick up the new token automatically.

Error state notes

  • Brick red is the only error color — used sparingly (dashed circle + the "!" glyph), not flooding the whole screen. Keeps the paper mood.
  • Three steps, numbered — concrete shell commands in monospace, shown in a warm card to feel "copy this."
  • Retry button fires a fresh auth check without reloading the app. If it still fails, it stays on this screen with a small subtle message: "still not seeing a token."
  • Same pattern reused for other fatal startup errors (SQLite inaccessible, port in use). Different glyph, same layout.

Modal vocabulary

  • Warm paper card slightly brighter than the page (#faf4e1) so it floats a touch. Sits over a dimmed paper scrim.
  • Label eyebrow in IBM Plex Mono at 9.5px, uppercase, muted — numbers each modal so you can reference them in code.

Mobile navigation

  • Top-right ··· icon opens the drawer from the right.
  • Drawer covers ~80% width, scrim behind dims the list. Tap scrim or × to close.
  • Two sections: main views (All issues, PRs) and actions (Quick Create, Settings).
  • Footer shows auth status, version, and the LAN-bound address so you can tell someone else the URL.

PR row anatomy

  • Status dot replaces checkbox — a filled ✓ circle for CI pass, a brick × for CI fail, a pulsing yellow dot for CI pending, a purple filled ✓ for merged.
  • Meta line: repo · #num · closes #X · CI status + age.
  • Sections renamed for PRs: ready to merge / in review / shipped (no "unassigned," no "in flight").
  • Swipe on a PR row goes to the PR detail (no assign action — PRs are always on a repo).

Priority & order

  • Three levels: high / normal / low. Default is normal. Local-only, never synced to GitHub.
  • Sort rule within a section: high first, then normal, then low. Ties break by updated-at.
  • No manual drag-to-reorder — changing priority is how you reorder. Simpler than a drag gesture, works on desktop too.
  • Visual marker on high-priority rows only: small ↑ glyph on the meta row (seen on the detail page in this frame). Normal and low have no marker — position in the list carries the signal.
  • Picker triggered from the detail page (tap the priority indicator in the meta row) and from the new-issue form (priority field below the repo field).

Decisions so far

  • Network model: app runs locally on a PC/Mac and binds to the LAN interface so phones on the same network can reach it directly. Remote access is user-managed via VPN. No auth shim in v1 — trust the LAN/VPN.
  • Priority default: new drafts start at normal.
  • Welcome copy: pulled back from "quiet place" to something more functional — "every issue, every repo. launch any of them with Claude Code."
  • Empty-list mark: replaced ✓ with ❧ (hedera, a classical typographic ornament marking "end of section"). More paper-notebook, less checkmark.
  • Section names: keeping unassigned / in focus / in flight / shipped for v1.

Screen coverage

  • Complete. Two audit agents compared the current app code, the spec, and the original mockup against this set. All essential v1 screens are now mocked: main list, issue detail, new issue, Claude launch (form + progress), first run, empty state, settings, quick create (3-step), PR detail, auth error, and the 5 small modal patterns (close / edit / labels / add repo / clone prompt).
  • Intentionally deferred to the design doc as text-only: 404 page, generic error boundary, loading skeletons, inline comment form (visible in issue detail), PR tab variant of the main list (trivial variant).
